1. Arquitectura General del Sistema
Un sistema moderno de HCE debe construirse bajo una arquitectura modular, cloud-native y orientada a servicios. La investigación académica (arXiv 2023-2025) y los proyectos open source de referencia convergen en una arquitectura de microservicios con capacidades offline para contextos de bajos recursos.

TENDENCIA 2025: La arquitectura cloud-native con contenedores (Docker/Kubernetes) y API-first se consolida como el estándar global. El estudio 'Redesigning EHR Systems for Developing Countries' (arXiv, 2023) propone además soluciones offline para zonas rurales sin conectividad.

2. Seguridad y Autenticación — Zero Trust Architecture
ACTUALIZACIÓN CRÍTICA 2025: La regla de seguridad HIPAA actualizada (publicada en diciembre 2024) eliminó la distinción entre especificaciones 'requeridas' y 'direccionables', haciendo obligatoria la segmentación de red y las medidas avanzadas de ciberseguridad. El costo promedio de una brecha en salud alcanzó USD 7.42 millones en 2025 (IBM Report). La arquitectura Zero Trust deja de ser una opción y se convierte en requisito regulatorio.

2.1 Modelo Zero Trust — 'Never Trust, Always Verify'
El paradigma Zero Trust reemplaza definitivamente el modelo perimetral tradicional. El NIST y las últimas guías HIPAA recomiendan explícitamente implementar principios Zero Trust para el control de acceso a ePHI.
Principio Zero Trust
Implementación en HCE
Microsegmentación
Segmentos aislados para: EHR, PACS, farmacia, laboratorio, dispositivos IoMT/IoT médicos.
Verificación continua
Re-evaluación dinámica de sesiones ante cambios de contexto: ubicación, dispositivo, comportamiento.
Mínimo privilegio
Cada rol accede únicamente a los datos necesarios para su función clínica.
MFA resistente a phishing
FIDO2/WebAuthn para roles privilegiados. TOTP como mínimo para todos.
Identidad como perímetro
Gobernanza centralizada de identidades con auditoría granular.
Tráfico cifrado interno
TLS mutuo (mTLS) entre microservicios. No confiar en tráfico interno no verificado.

2.2 Autenticación — OAuth 2.0 + OpenID Connect
Mecanismo
Aplicación
Authorization Code + PKCE
Para SPAs y apps móviles. Flujo más seguro para usuarios finales.
Client Credentials
Para integraciones sistema-a-sistema: laboratorios, sistemas gubernamentales, seguros.
Device Authorization
Para dispositivos médicos IoT con entrada de usuario limitada.
JWT (Access Tokens)
Vida corta (15-30 min). Claims clínicos: rol, especialidad, unidad, establecimiento.
Refresh Tokens rotativos
Renovación segura de sesiones con rotación automática y revocación.
Keycloak (recomendado)
IDP open source. Soporta SAML, OIDC, federación con LDAP/AD gubernamentales.

3. Gestión Integral del Paciente
3.1 Registro Demográfico
El registro demográfico debe cumplir con los estándares HL7 FHIR Patient Resource e incluir campos requeridos por la normativa local (Argentina: DNI; internacional: MPI - Master Patient Index).
Campo
Descripción
Identificación
Nombre completo, documento nacional (DNI/Pasaporte/CI), número de historia clínica (MRN).
Datos demográficos
Fecha de nacimiento, sexo biológico, género autopercibido (LGBTQ+ inclusivo), nacionalidad.
Contacto
Teléfono principal y alternativo, email, dirección completa con geolocalización opcional.
Contacto de emergencia
Nombre, relación, teléfono. Múltiples contactos ordenados por prioridad.
Cobertura médica
Obra social/prepaga, número de afiliado, plan, vigencia. Múltiples coberturas.
Datos clínicos base
Grupo sanguíneo, factor RH, donante de órganos (consentimiento).
Datos de riesgo
Alergias conocidas (integradas con alertas en prescripción), discapacidades.
Trazabilidad
Registro de cambios demográficos con fecha, usuario y motivo del cambio.

3.2 Historia Clínica Integral — Estructura FHIR
La historia clínica debe modelarse siguiendo los recursos FHIR R4 estándar, permitiendo interoperabilidad nativa con otros sistemas de salud.
Sección
Contenido
Antecedentes personales
Enfermedades crónicas, cirugías previas, hospitalizaciones, traumatismos.
Antecedentes familiares
Enfermedades hereditarias relevantes (cardiopatías, diabetes, cáncer, etc.).
Hábitos y estilo de vida
Tabaquismo, alcohol, actividad física, alimentación, trabajo (riesgos laborales).
Medicación actual
Listado activo de medicamentos con dosis, frecuencia, prescriptor y fecha inicio.
Alergias e intolerancias
Sustancia, tipo de reacción, severidad, fecha de detección. Integrada con SNOMED CT.
Esquema de vacunación
Vacunas aplicadas, lote, fecha, profesional aplicador. Compatible con registro nacional.
Evolución médica
Consultas, hospitalizaciones, procedimientos, interconsultas con trazabilidad completa.
Resultados de laboratorio
Histórico cronológico con gráficos evolutivos y alertas de valores críticos.
Imágenes diagnósticas
Integración DICOM/PACS. Reportes radiológicos vinculados.

3.3 Gestión de Episodios Clínicos
El sistema debe gestionar episodios diferenciados por tipo de atención, cada uno con su propio ciclo de vida y flujo de trabajo clínico:
Consulta ambulatoria: el tipo más frecuente. Flujo: admisión → triaje → consulta → documentación → alta.
Episodio de hospitalización: admisión → evaluación → tratamiento → epicrisis → alta con seguimiento.
Urgencia/Emergencia: triaje ESI/Manchester → atención → resolución. Integración con urgencias.
Telemedicina: videoconsulta → documentación → prescripción remota → seguimiento.
UCI/Cuidados Críticos: monitoreo continuo, ventilación, protocolos específicos, scoring (APACHE, SOFA).
Cirugía: preoperatorio → intraoperatorio → postoperatorio → alta quirúrgica.


4. Documentación Clínica — IA Ambiental y Notas Estructuradas
REVOLUCIÓN 2024-2025: Los AI Medical Scribes (escribas IA ambientales) se consolidan como la transformación más significativa en documentación clínica. Un estudio del Permanente Medical Group (2025) con 7.260 médicos reportó un ahorro de 15.791 horas anuales gracias a scribes IA integrados con el EHR. Al menos 60 vendors operan en este espacio. Berta, un scribe open source, fue desplegado en 198 médicos de urgencias con 22.148 sesiones clínicas generadas.

4.1 Notas SOAP — El Estándar Clínico Universal
Componente SOAP
Contenido Recomendado
S — Subjective
Motivo de consulta narrado por el paciente. Síntomas, duración, intensidad (EVA). Historia de la enfermedad actual.
O — Objective
Examen físico: signos vitales, hallazgos por aparatos. Resultados de laboratorio e imágenes del episodio.
A — Assessment
Diagnóstico principal (CIE-10/ICD-11/SNOMED CT) y diagnósticos secundarios. Diagnósticos diferenciales.
P — Plan
Tratamiento prescripto, estudios solicitados, derivaciones, recomendaciones, fecha de seguimiento.

4.2 IA Ambiental para Documentación — Estado del Arte 2025
La documentación clínica asistida por IA (AI Medical Scribes) es la funcionalidad diferenciadora más impactante del bienio 2024-2025:
Capacidad IA
Evidencia y Tecnología
Ambient Listening
Captura pasiva de la conversación médico-paciente sin interrumpir la consulta. WhisperX, Parakeet (NVIDIA), Amazon Transcribe.

Reducción de burnout
El 90% de los médicos de atención primaria de Mass General Brigham solicitaron acceso al scribe IA.

8. Interoperabilidad — HL7 FHIR y Estándares
ESTADO ACTUAL (2025): FHIR R4 es el estándar global más adoptado (22 de 38 organizaciones según la encuesta State of FHIR 2024). R5 está en adopción limitada. FHIR R6 está en proceso de ballot con fecha prevista para 2026. El framework LLM+FHIR via Model Context Protocol (MCP-FHIR) emerge como nueva forma de integrar IA con datos clínicos estructurados (arXiv 2025).

10. Inteligencia Artificial y Soporte de Decisiones Clínicas
ESTADO DEL ARTE 2025: Los LLMs (Large Language Models) integrados en los EHR son la frontera más activa de la investigación. El sistema LATCH (2026) reproduce hallazgos de 20 estudios publicados en 3-15 minutos por hipótesis, automatizando la investigación clínica.

10.3 Consideraciones Éticas y de Gobernanza de IA
La implementación de IA en HCE requiere un marco de gobernanza robusto. La Unión Europea (AI Act 2024) clasifica los sistemas de IA en salud como 'alto riesgo', requiriendo registro, documentación técnica y evaluación de conformidad:
Explicabilidad (XAI): el clínico debe entender por qué la IA generó una alerta o recomendación.
Supervisión humana: la IA asiste, no decide. El profesional de salud siempre tiene la última palabra.
Datos de entrenamiento: sin sesgos de género, etnia, nivel socioeconómico. Auditoría periódica.
Privacidad: uso exclusivo de datos anonimizados o sintéticos para entrenamiento. Nunca datos reales sin consentimiento.
Validación clínica: los modelos deben validarse en la población local antes de despliegue en producción.
Trazabilidad: registro de qué modelo, qué versión y qué datos generaron cada recomendación clínica.

El ecosistema de Historia Clínica Electrónica está atravesando la transformación más profunda de su historia, impulsada por tres fuerzas convergentes: la madurez de FHIR como estándar global, la irrupción de los LLMs en documentación clínica y la consolidación del modelo Zero Trust en seguridad.

Las 5 Mega-Tendencias que definen el HCE de próxima generación:

Tendencia
Estado Actual y Proyección

2. FHIR como lingua franca
FHIR R4 dominante. R6 en draft para 2026. MCP-FHIR permite a los LLMs razonar sobre datos clínicos en tiempo real. El EHR se convierte en plataforma abierta de datos.

Los proyectos open source más maduros (OpenEMR 7.x, OpenMRS/Bahmni, Ozone HIS) demuestran que la modularidad, la interoperabilidad FHIR y la comunidad activa son los factores clave de supervivencia y evolución. Un nuevo sistema HCE debe nacer con estas bases para ser relevante en el horizonte 2025-2030.

# Blueprint operativo de agentes para la HCE

## 1. Objetivo
Operar una HCE desde el día 1 con un único super admin humano y un conjunto de agentes de IA especializados, cada uno con responsabilidades acotadas, permisos mínimos y trazabilidad completa.

La arquitectura debe sostener interoperabilidad FHIR, autenticación con Keycloak/OIDC, despliegue contenedorizado y una capa de IA conectada mediante MCP/FHIR, no mediante acceso irrestricto al sistema [web:2][web:3][web:6][web:8][web:12][web:16].

## 2. Principios
- Un solo orquestador con visión global.
- Agentes por dominio, no un agente generalista.
- Permisos mínimos por tarea.
- Toda acción sensible queda auditada.
- IA como asistente, no como autoridad clínica.
- La interoperabilidad y el control de acceso se diseñan desde el inicio [web:2][web:8][web:12][web:16].

## 3. Stack de agentes

### 3.1 Agente Orquestador
**Rol:** recibe objetivos del super admin, los divide, asigna trabajo y consolida resultados.

**Prompt base:**
“Eres el orquestador central del proyecto HCE. Tu tarea es descomponer objetivos en tareas concretas, asignarlas al agente correcto, detectar dependencias, priorizar por valor clínico y riesgo, y consolidar decisiones. Nunca asumas permisos fuera del plan. Mantén trazabilidad y estado actual del proyecto.”

**Entradas:**
- Objetivos del super admin.
- Resultados de otros agentes.
- Estado del backlog.

**Salidas:**
- Tareas.
- Prioridades.
- Resúmenes.
- Decisiones consolidadas.

**Permisos:**
- Lectura global.
- Escritura sobre tareas y prioridades.
- Sin acceso directo a datos clínicos productivos.

### 3.2 Agente Arquitecto
**Rol:** define la arquitectura técnica y los límites de dominio.

**Prompt base:**
“Eres el arquitecto principal de una HCE. Diseña una arquitectura modular, API-first, interoperable con FHIR y preparada para crecer sin deuda técnica innecesaria. Evalúa trade-offs, delimita dominios y propone decisiones explícitas y justificadas.”

**Salidas:**
- Decisiones de arquitectura.
- Diagramas.
- Contratos internos.
- Roadmap técnico.

**Permisos:**
- Escritura de decisiones técnicas.
- Lectura de requerimientos y restricciones.
- Sin tocar seguridad global ni lógica clínica.

### 3.3 Agente FHIR/MCP
**Rol:** traduce procesos clínicos a recursos FHIR y herramientas MCP.

**Prompt base:**
“Eres el especialista en interoperabilidad clínica. Convierte flujos de atención en recursos FHIR, diseña mapeos consistentes y define herramientas MCP seguras para que la IA interactúe con datos clínicos estructurados. Prioriza compatibilidad, claridad semántica y control de permisos.”

**Salidas:**
- Mapeos FHIR.
- Contratos MCP.
- Esquemas de recursos.
- Reglas de interoperabilidad.

**Permisos:**
- Escritura sobre especificaciones FHIR/MCP.
- Lectura del dominio clínico.
- Sin acceso a credenciales ni políticas globales.

### 3.4 Agente Seguridad y Compliance
**Rol:** protege identidad, acceso, auditoría y cumplimiento.

**Prompt base:**
“Eres el responsable de seguridad y cumplimiento de una HCE crítica. Diseña controles Zero Trust, MFA, auditoría inmutable, segmentación, gestión de secretos y políticas de mínimo privilegio. Reporta riesgos, brechas y medidas concretas. No permitas atajos que comprometan la seguridad.”

**Salidas:**
- Políticas.
- Alertas.
- Recomendaciones.
- Hallazgos de riesgo.

**Permisos:**
- Lectura amplia.
- Escritura de políticas y hallazgos.
- Sin alterar reglas asistenciales.

### 3.5 Agente de Producto Clínico
**Rol:** prioriza el valor asistencial real.

**Prompt base:**
“Eres el traductor de la práctica clínica al producto. Tu tarea es convertir necesidades asistenciales en historias de usuario y flujos concretos, priorizando lo que resuelve problemas reales. El foco es reducir fricción clínica y evitar funciones innecesarias.”

**Salidas:**
- Historias de usuario.
- Criterios de aceptación.
- Priorización por valor clínico.

**Permisos:**
- Escritura funcional.
- Lectura de flujos.
- Sin acceso a despliegue o seguridad.

### 3.6 Agente UX/HCE
**Rol:** diseña la experiencia de uso.

**Prompt base:**
“Eres responsable de la experiencia de uso de una HCE. Diseña pantallas rápidas, claras y clínicas. Minimiza clics, evita ruido visual y optimiza el trabajo de médicos, enfermería, recepción y pacientes.”

**Salidas:**
- Wireframes.
- Patrones de navegación.
- Reglas de interacción.
- Microcopy.

**Permisos:**
- Escritura UX.
- Lectura funcional.
- Sin acceso a datos clínicos completos.

### 3.7 Agente de Integraciones
**Rol:** conecta la HCE con sistemas externos.

**Prompt base:**
“Eres el especialista en integraciones de salud. Debes conectar la HCE con LIS, PACS, DICOM, aseguradoras, organismos públicos y servicios de notificación. Toda integración debe ser robusta, trazable y compatible con contratos explícitos.”

**Salidas:**
- Conectores.
- Adaptadores.
- Esquemas de integración.
- Manejo de errores.

**Permisos:**
- Escritura sobre conectores.
- Lectura de contratos externos.
- Sin acceso al núcleo completo.

### 3.8 Agente QA/Test
**Rol:** valida el sistema.

**Prompt base:**
“Eres el agente de calidad de una HCE crítica. Genera pruebas funcionales, de regresión y de borde. Detecta errores, inconsistencias y riesgos antes de liberar. No apruebes nada sin evidencia verificable.”

**Salidas:**
- Casos de prueba.
- Reportes de fallos.
- Evidencia de cobertura.
- Riesgos de release.

**Permisos:**
- Lectura amplia.
- Escritura de tests y reportes.
- Sin capacidad de liberar a producción.

### 3.9 Agente DevOps/Release
**Rol:** opera el sistema.

**Prompt base:**
“Eres el responsable de despliegue, monitoreo y continuidad operativa. Debes preparar pipelines, releases, rollback y observabilidad. Prioriza estabilidad, trazabilidad y recuperación rápida.”

**Salidas:**
- Pipelines.
- Releases.
- Métricas.
- Alertas.

**Permisos:**
- Escritura operativa.
- Lectura de artefactos.
- Sin modificar negocio clínico.

## 4. Permisos por agente
| Agente | Lectura | Escritura | Límite |
|---|---|---|---|
| Orquestador | Todo el proyecto | Tareas, prioridades, estado | No toca clínica ni seguridad global |
| Arquitecto | Specs y restricciones | Decisiones técnicas | No altera lógica clínica |
| FHIR/MCP | Dominio clínico | Contratos FHIR/MCP | No maneja secretos |
| Seguridad | Todo | Políticas y hallazgos | No cambia negocio |
| Producto Clínico | Flujos y requerimientos | Historias de usuario | No despliega |
| UX/HCE | Requerimientos | Wireframes y UX | No accede a datos completos |
| Integraciones | Contratos externos | Conectores | No decide arquitectura global |
| QA/Test | Artefactos | Tests y reportes | No libera producción |
| DevOps/Release | Build y runtime | Pipelines y deploy | No modifica reglas clínicas |

## 5. Herramientas MCP sugeridas
- **MCP-FHIR:** recursos clínicos estructurados.
- **MCP-Docs:** notas, consentimientos y adjuntos.
- **MCP-Security:** auditoría, permisos y eventos.
- **MCP-Integrations:** LIS, PACS, SISA, seguros y farmacias.
- **MCP-Planning:** backlog, roadmap y tareas.

## 6. Flujo operativo
1. El super admin define el objetivo.
2. El Orquestador lo descompone.
3. Arquitecto, Seguridad y FHIR/MCP preparan la base.
4. Producto Clínico valida el flujo.
5. UX e Integraciones lo convierten en sistema usable.
6. QA prueba.
7. DevOps libera.
8. El Orquestador consolida estado y próximos pasos.

## 7. Base técnica recomendada
La evidencia pública favorece una base con FHIR, Keycloak y Kubernetes, más SMART on FHIR para acceso clínico seguro. También hay proyectos abiertos de scribe clínico y MCP/FHIR que muestran viabilidad práctica para IA sanitaria [web:2][web:3][web:6][web:8][web:10][web:12][web:15][web:16].

**Stack sugerido:**
- Frontend: React + TypeScript.
- Backend: API modular.
- FHIR server: HAPI FHIR o equivalente.
- Identity: Keycloak.
- Orquestación: Kubernetes.
- Mensajería: RabbitMQ o Kafka.
- Base de datos: PostgreSQL.
- Observabilidad: Prometheus + Grafana.
- Acceso clínico: SMART on FHIR + OIDC [web:2][web:8][web:12].

## 8. IA clínica: uso correcto
Sí:
- Resúmenes de paciente.
- Borradores de SOAP.
- Extracción de entidades.
- Sugerencia de códigos.
- Scribe ambiental.
- Consulta sobre recursos estructurados [web:3][web:4][web:10][web:15].

No:
- Escritura directa sin revisión.
- Decisión clínica autónoma.
- Acceso irrestricto a toda la historia.
- Mezclar administración con clínica sin separación [web:3][web:15].

- Analítica avanzada [web:3][web:4][web:10][web:15].

## 10. Criterios de éxito
- Cada agente tiene un dominio claro.
- El orquestador central controla el flujo.
- La identidad se resuelve con Keycloak u otro IdP robusto.
- Los datos clínicos viven en FHIR.
- La IA opera sobre MCP/FHIR.
- Todo queda auditado.
- El sistema puede crecer sin romper seguridad ni usabilidad [web:2][web:8][web:12][web:16].
